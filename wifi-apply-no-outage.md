# Applying WiFi changes without a full wireless outage

> **Verified live 2026-06-04.** Goal: let the webui create/edit a WiFi network
> **without** the global Wi-Fi outage caused by `restart_wireless`. SSIDs/BSSIDs/keys
> are redacted throughout.

## The problem with the current apply path

The webui's SDN apply (`net_apply_one`) ends in `restart_wireless;restart_sdn`.
`restart_wireless()` (`rc/lan.c:6569` in the firmware source) is a **full teardown**:
it sets `wlready=0`, toggles MLO, tears down AMAS, and restarts the **driver + all
four radios + every bridge + the SDN** â€” i.e. **all Wi-Fi drops for ~10â€“15 s**
(CLAUDE.md notes the ~5â€“15 s drop). There is **no lighter built-in path**: stock
ASUSWRT only ever uses `restart_wireless`, and only calls `hostapd_cli` for WPS /
`get_config`, never for a config reload (verified by grep across `rc/` + `shared/`).

## Live process & control model (the enabler)

```
hostapd /tmp/wl0_hapd.conf -B      # one hostapd per RADIO
hostapd /tmp/wl1_hapd.conf -B      #   each owns that band's primary + virtual BSS
hostapd /tmp/wl2_hapd.conf -B
hostapd /tmp/wl3_hapd.conf -B
eapd                               # one shared
```
Per-**BSS** hostapd control sockets exist under `/var/run/hostapd/`:
`wl0 wl0.1 wl1 wl2 wl2.1 wl3 wl3.2 wl3.3 wl3.4`. `hostapd_cli`, `hostapd`,
`wpa_cli` are all present.

Because hostapd is **per-radio**, the blast radius of any reload is at most **one
band** â€” never all four.

## What can change with ZERO outage (pure runtime)

Verified live; siblings and other radios stay up, no restart:

| Change | Command | Outage |
|---|---|---|
| SSID (primary or virtual) | `wl -i <bss> ssid "<name>"` | none â€” instant (E5/E8) |
| Hide / unhide | `wl -i <bss> closed 0\|1` then `hostapd_cli -i <bss> update_beacon` | none |
| Enable / disable a BSS | `wl -i <bss> bss up\|down` or `hostapd_cli -i <bss> disable\|enable` | none (only that BSS) |
| Move a BSS to another VLAN | `brctl delif <oldbr> <bss> ; brctl addif <newbr> <bss>` | none |

`hostapd_cli` verbs available on this build: `enable`, `disable` (per interface),
`update_beacon`, `set`/`get`, `raw`, `wps_*`. **No `reconfigure`** verb.

## What needs a ONE-BAND reload (not global)

Changing **WPA security** (passphrase / auth mode) or **adding a brand-new BSS**
requires hostapd to re-read its config for that radio. Two levers:

- **`SIGHUP` to that radio's hostapd** â€” verified: HUP'ing `wl1`'s hostapd left
  `wl0/wl2/wl3` (and all their user nets) **up**. Reloads only that band's BSS.
- Rewrite `/tmp/wlX_hapd.conf` for the one radio, then HUP it.

So the worst case is a brief blip on a **single band**, versus the all-radio outage
of `restart_wireless`.

## Why this is hostapd-direct (and needs the firmware work)

All of the above **bypasses `cfg_server`**. As long as `cfg_server`/`mtlancfg` is
live it will regenerate `/tmp/wlX_hapd.conf` and re-drive `restart_wireless` on its
own triggers (boot, AiMesh sync, any `restart_wireless` someone else calls),
**reverting** hand-applied runtime changes. Persistence therefore needs `cfg_server`
removed/neutralized â€” the same firmware prerequisite as
[main-wifi-admin-lan.md](main-wifi-admin-lan.md) and CLAUDE.md Phase 3:
*"once cfg_server is removed, nothing regenerates hostapd anymore â†’ the
hostapd-direct approach becomes viable again."*

## Proposed apply model (webui owns hostapd, post-cfg_server)

1. webui maintains `/tmp/wlX_hapd.conf` per radio from its own config store.
2. **Edit SSID / visibility / enable-disable / VLAN** â†’ apply with `wl` + `brctl` +
   `hostapd_cli update_beacon` â†’ **zero outage**.
3. **Edit security / create a new BSS** â†’ rewrite the affected radio's conf, `SIGHUP`
   that one hostapd â†’ **one-band blip only**.
4. **Never call `restart_wireless`** for routine edits.

This unifies the three goals: no hidden admin SSID, first net adopts the primary on
its VLAN, and outage-free edits â€” all delivered by the webui owning hostapd directly
once the firmware retires `cfg_server`.

## Prototype: webui-owned hostapd config generator

`src/cgi-bin/lib/hapd_gen.sh` emits a driver-compatible per-radio
`/tmp/wlX_hapd.conf` from the webui's own model (the clean successor to the
Phase-B4-deleted generator). `hapd_gen.sh selftest` renders a sample with no device.

**Live validation (2026-06-04, idle radio wl1):** a conf generated by `hapd_gen.sh`
(WPA2-PSK, `bridge=br20`) was handed to the **open** `hostapd`; debug showed it
initialise the **nl80211 driver on wl1, set AP mode, subscribe to mgmt frames, and
reach `AP-ENABLED`** â€” proof the open hostapd brings up an AP on this Broadcom radio
from our config. Caveat: a bare `hostapd -B` launched *alongside the running stock
stack* (eapd + the BCM hostapd-monitor that respawns the stock conf) does not survive
â€” the monitor/`cfg_server` fight it. So the generator is correct, but using it as the
authoritative path needs the stock generator + `cfg_server` out of the way (firmware
patch 0030). Until then `hapd_gen.sh` is a **standby/test** path, gated off.

## Webui apply path (pre-staged)

`net_apply_one` (`src/cgi-bin/lib/networks.sh`) now branches: an **edit** of an
existing BSS uses `net_sdn_scoped_apply` (per-radio `rc restart_wireless_unit` +
`restart_sdn`, no global outage) **iff** the firmware capability is present *and*
`nvram webui_scoped_apply=1`; otherwise it falls back to the full
`restart_wireless;restart_sdn`. Creation/deletion always uses the full path (new VLAN
bridge composition). Default behaviour is unchanged until the patched firmware is
flashed and the flag is set.

## Firmware-source notes (read-only study of `../gt-be98-firmware`)

- `restart_wireless()` = `rc/lan.c:6569` â€” the heavy teardown; no lighter variant.
- The BSS list **and bridge membership** fed to the closed `hostapd_config_be.o`
  generator come from `get_all_lanifnames_list()` in
  `shared/wlif_utils_ax.c` (patched by `0025` for MAINFH via
  `hapd_exclude_ifnames`). This is why the primary's `bridge=br0` can't be moved by
  `lan_ifnames` (main-wifi doc, E7): the closed generator computes it from the SDN
  model, and the head `interface=` line can't be excluded the way a secondary BSS can.
